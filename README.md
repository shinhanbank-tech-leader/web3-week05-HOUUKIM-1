## 1. EOA

### 1-1. EOA

이더리움에는 두 가지 계정 유형이 있다. EOA는 그 중 하나로, 개인키를 직접 보유한 사람이 제어하는 외부 소유 계정이다. 나머지 하나는 스마트컨트랙트 코드로 동작하는 컨트랙트 계정이다.

EOA의 동작 방식은 단순하다.

```
개인키로 트랜잭션에 서명 → 서명된 트랜잭션이 네트워크에 전파 → 검증 후 상태 변경
```

이 구조는 이더리움 출시 이후 지금까지 검증된 방식이며, 모든 dApp과 툴의 기본 단위다.

### 1-2. EOA의 장점

- **단순한 구조**: 스마트컨트랙트 없이 개인키만으로 동작하므로 컨트랙트 버그, 취약점 공격 위험이 없다.
- **낮은 가스비**: 중간 계층 없이 직접 트랜잭션을 발생시키므로 오버헤드가 최소화된다.
- **완전한 호환성**: 이더리움의 모든 서비스, 툴, 거래소가 EOA 기준으로 설계되어 있다.
- **검증된 안정성**: 수십억 달러 규모의 자산이 수년간 EOA로 운용되어 왔다.

### 1-3. EOA의 한계 (불편함, 위험성)

그러나 EOA는 대중화 측면에서 구조적인 한계를 갖는다.

**1 시드 구문·개인키 분실 = 자산 영구 소실**

EOA의 가장 큰 위험은 키 관리다. 12~24개 단어로 이루어진 시드 구문을 잃어버리거나 도난당하면 계정 복구 방법이 전혀 없다. 은행이라면 신분증 한 장으로 계정을 복구할 수 있지만, EOA는 그 어떤 신뢰 기관도 개입할 수 없는 구조다.

**2 가스비를 항상 ETH로 직접 지불해야 함**

EOA로 어떤 트랜잭션이든 발생시키려면 지갑에 ETH가 있어야 한다. 예를 들어 USDC만 보유한 사용자는 단순 토큰 전송 한 번을 위해서도 ETH를 별도로 구매해야 한다. 이는 신규 사용자 진입 장벽으로 직결된다.

**3 단일 서명 구조 - 다중 승인 불가**

개인키 하나가 모든 권한을 갖는다. 은행의 이중·삼중 승인 체계나 한도 분리 같은 내부 통제 로직을 EOA 수준에서는 구현하기 어렵다.

**4 트랜잭션 배치 처리 불가**

EOA는 한 번에 하나의 트랜잭션만 처리한다. 여러 작업을 하나로 묶는 배치 처리를 위해서는 별도 스마트컨트랙트가 필요하다.

**5 개인적 의견**

EOA 구조는 마치 고객이 직접 원장에 접근하는 것과 같다. 은행 시스템은 고객과 원장 사이에 수많은 통제 계층(인증, 승인, 한도 관리, 감사 로그)을 두지만, EOA는 이 모든 중간 계층이 없다. 보안성과 자기주권성이라는 이점이 있지만, 대규모 고객 서비스에 적용하기엔 복구 불가능성과 단일 장애점(키 분실) 문제가 크다.

## 2. ERC-4337

### 2-1. 계정 추상화란

ERC-4337은 2021년 이더리움 개발자들이 제안한 표준으로, 지갑 자체를 스마트컨트랙트로 구현하는 '계정 추상화'를 실현한다. 핵심 아이디어는 **'서명과 인증 로직을 코드로 자유롭게 정의할 수 있다'** 는 것이다. 기존 EOA가 항상 ECDSA 서명만 써야 했다면, ERC-4337은 서명 방식, 승인 조건, 수수료 정책을 프로그래밍으로 정의할 수 있다.

### 2-2. 핵심 구성 요소

| 구성 요소 | 역할 | 비유 |
|---|---|---|
| **UserOperation** | 기존 트랜잭션 대신 사용되는 새 객체. 실행할 작업과 서명이 담긴다. | 전자 결재 문서 |
| **Bundler** | 여러 UserOperation을 묶어 EntryPoint에 제출하는 노드 | 전문 처리 서버(MCI) |
| **EntryPoint** | 모든 UserOperation을 검증·실행하는 중앙 컨트랙트 | 공통 처리 허브 |
| **Paymaster** | 가스비를 대신 지불하거나 다른 토큰으로 납부 가능하게 해주는 컨트랙트 | 가스비 대납 서비스 |
| **Smart Account** | 사용자의 실제 지갑 컨트랙트. 서명 로직·권한 정책이 담겨 있음 | 스마트 지갑 |

### 2-3. ERC-4337이 가능하게 한 것들

- **소셜 로그인 지갑**: 구글·애플 계정으로 지갑 생성 가능. 시드 구문 필요 없음.
- **가스비 대납(Sponsorship)**: 서비스 제공자가 사용자 가스비를 대신 지불. 신규 사용자도 ETH 없이 첫 거래 가능.
- **다중 서명(MFA) 지원**: 2-of-3 멀티시그, 생체인증 등 다양한 인증 방식 적용 가능.
- **배치 트랜잭션**: 여러 작업을 하나의 UserOperation으로 묶어 처리. 복잡한 DeFi 거래 단순화.
- **계정 복구**: 신뢰 가능한 주소(guardian)를 통한 키 복구 기능 구현 가능.
- **세션 키**: 특정 dApp에 제한된 권한을 부여하는 일회성 키 발급 가능.

### 2-4. ERC-4337의 한계와 새로운 운영 고민

사용성이 좋아진 만큼, 운영·통제 측면에서 새롭게 고민해야 할 것들이 생긴다.

- **높은 가스비**: Bundler, EntryPoint 레이어를 거치면서 동일 작업 기준 가스비가 EOA 대비 증가한다.
- **복잡한 책임 구조**: 장애 발생 시 Bundler 문제인지, EntryPoint 문제인지, Paymaster 문제인지 추적이 복잡해진다.
- **Paymaster 의존성**: 가스비를 대납하는 Paymaster가 중단되거나 자금이 소진되면 서비스 전체가 멈출 수 있다.
- **복구 권한 관리**: guardian 기능이 잘못 설정되면 제3자가 계정을 탈취할 수 있다. 복구 권한 자체를 누가 관리하느냐가 새로운 리스크가 된다.
- **스마트컨트랙트 버그 위험**: 지갑 로직이 코드로 구현되므로, 컨트랙트 취약점이 곧 자산 탈취로 이어질 수 있다.

### 개인적 의견

ERC-4337은 은행의 '계정 관리 구조'와 흥미롭게 닮아있다. 은행도 고객 계좌(Smart Account)와 처리 채널(Bundler), 공통 처리 시스템(EntryPoint), 수수료 정책(Paymaster)이 분리되어 있다. 다만 은행은 이 모든 것이 중앙집중적으로 통제되지만, ERC-4337은 탈중앙화된 스마트컨트랙트로 구현된다는 점이 다르다. 결국 사용성 향상은 '탈중앙화된 은행 계층'을 만드는 것이고, 그에 따른 운영 리스크와 책임 소재 관리가 핵심 과제가 된다.

## 3. 비교

### 3-1. EOA와 ERC-4337 비교

| 구분 | EOA | ERC-4337 |
|---|---|---|
| 제어 방식 | 개인키 직접 서명 | 스마트컨트랙트 로직 |
| 키 분실 시 | 자산 영구 소실 | guardian을 통한 복구 가능 |
| 가스비 납부 | 반드시 ETH 직접 보유 | Paymaster 통해 대납·토큰 납부 가능 |
| 서명 방식 | ECDSA 고정 | 자유롭게 커스터마이징 가능 |
| 다중 승인 | 불가 | 멀티시그·MFA 구현 가능 |
| 배치 처리 | 불가 (1건씩) | 가능 (여러 작업 묶기) |
| 가스 비용 | 낮음 (직접 처리) | 높음 (레이어 오버헤드) |
| 보안 위험 | 키 분실·도용 | 컨트랙트 버그·Paymaster 장애 |
| 적합한 사용자 | 개발자·고급 사용자 | 일반 사용자·대중화 서비스 |

### 3-2. 은행 인증·계정 관리와의 비교

은행 IT 실무자 입장에서 두 방식을 은행 시스템과 비교하면 다음과 같은 통찰을 얻을 수 있다.

| 관점 | 은행 시스템 | EOA | ERC-4337 |
|---|---|---|---|
| 인증 방식 | ID/PW + OTP + 생체인증 | 개인키 서명 (단일) | 커스텀 서명 (MFA 가능) |
| 계정 복구 | 신분증 제출 후 복구 | 불가 | Guardian 설정 시 가능 |
| 승인 규칙 | 이중·삼중 승인 체계 | 단일 키가 모든 권한 | 컨트랙트로 승인 정책 정의 |
| 수수료 주체 | 은행이 내부 처리 | 사용자가 ETH로 직접 납부 | Paymaster가 대납 가능 |
| 장애 책임 | 은행이 SLA 책임 | 개인 책임 (복구 불가) | 컨트랙트·Bundler 책임 복잡 |
| 감사 로그 | 중앙 DB에 완전 보관 | 블록체인에 영구 기록 | 블록체인에 영구 기록 |
| 접근 통제 | 역할 기반 권한 관리 | 불가 | 컨트랙트로 권한 세분화 가능 |

주목할 점은 ERC-4337이 은행의 계정 관리 구조에 상당히 근접해 있다는 것이다. 은행은 오래전부터 '사용자는 간단하게, 내부는 정교하게' 설계해왔다. ERC-4337은 블록체인에서 같은 방향을 추구한다. 다만 은행은 신뢰받는 중앙기관이 책임을 지는 반면, ERC-4337은 코드와 컨센서스가 그 역할을 대체한다는 근본적인 차이가 있다.

## 4. 트릴레마

### 4-1. 트릴레마란

블록체인 트릴레마는 비탈릭 부테린이 제시한 개념으로, 블록체인이 동시에 달성하기 어려운 세 가지 속성을 가리킨다.


| 속성 | 의미 | 희생하면 |
|---|---|---|
| **보안성 (Security)** | 공격자가 네트워크를 조작하기 어렵게 만드는 성질 | 해킹·이중지불 취약 |
| **탈중앙화 (Decentralization)** | 특정 주체가 네트워크를 통제하지 못하는 성질 | 검열·조작 가능 |
| **확장성 (Scalability)** | 빠른 처리 속도와 낮은 비용으로 대량 거래 처리 가능한 성질 | 대중화 불가 |

예를 들어 보안성과 탈중앙화를 모두 갖추려면 수천 개의 노드가 합의를 이뤄야 해서 처리 속도가 느려진다. 반면 처리 속도를 높이려면 검증 노드 수를 줄이거나 중앙화해야 한다.

### 4-2. 왜 트릴레마가 중요한가

트릴레마는 단순한 기술 문제가 아니다. 블록체인이 실제 금융 서비스에 적용될 수 있느냐를 결정하는 핵심 제약이다.

- 현재 이더리움 메인넷: 초당 약 **15~30건** 처리
- VISA: 초당 약 **24,000건** 처리

현재 구조로는 글로벌 금융 인프라를 대체하는 것이 물리적으로 불가능하다. 국내 금융망도 마찬가지로, 대외계 시스템은 하루에도 수백만 건의 파일전송과 전문을 처리하는데, 이더리움 메인넷의 처리 능력으로는 감당이 안 된다.

### 4-3. 트릴레마 해결을 위한 기술적 접근

#### 1. Layer 2 (L2) - 현재 가장 현실적인 해법

L2는 메인넷(L1) 위에 별도 처리 레이어를 올려, L1의 보안성과 탈중앙화는 유지하면서 확장성을 확보하는 방식이다.

| L2 방식 | 원리 | 특징 |
|---|---|---|
| **Optimistic Rollup** | 일단 유효하다고 가정하고 처리, 이의 제기 기간 운영 | 구현 단순, 출금에 7일 소요 (Arbitrum, Optimism) |
| **ZK Rollup** | 수학적 영지식 증명으로 유효성 즉시 검증 | 빠른 최종성, 복잡한 구현 (zkSync, StarkNet) |
| **State Channel** | 두 당사자 간 오프체인 채널 개설, 최종 결과만 온체인 기록 | 마이크로페이먼트에 적합 (Lightning Network) |

#### 2. 샤딩(Sharding)

이더리움 2.0의 핵심 기술로, 블록체인을 여러 조각(shard)으로 나눠 병렬 처리하는 방식이다. 현재 이더리움은 L2와 병행하는 Danksharding 방향으로 개발 중이다.

#### 3. ERC-4337과 L2의 결합

ERC-4337 기반 스마트 지갑은 L2 위에서 동작할 때 시너지가 극대화된다. L2의 낮은 가스비로 ERC-4337의 오버헤드를 상쇄하고, 스마트 지갑의 사용성 개선이 L2의 대중화를 이끈다. 이 조합이 현재 Web3 대중화의 가장 유력한 경로로 꼽힌다.

#### 4. 트릴레마와 금융 시스템의 공통점
트릴레마는 사실 금융 시스템도 늘 직면해온 문제다. 은행은 보안성(인증·암호화)과 가용성(24시간 운영)을 갖추면서도 처리 속도(확장성)를 위해 중앙화된 구조를 선택했다. 블록체인은 탈중앙화를 포기하지 않으면서도 나머지 두 가지를 달성하려는 시도다. L2는 그 타협점의 현재 최선이라 볼 수 있다.

## 5. 현실적인 구조

### 5-1. 순수 퍼블릭 체인 도입은 현재 시기상조

이더리움 메인넷을 그대로 금융 서비스에 적용하는 것은 여러 면에서 현실적이지 않다. 처리 속도 한계, 규제 불확실성, 개인정보 온체인 기록 문제, 가스비 변동성 등이 장벽이다.

### 5-2. 현실적 구조 제안

대외계 개발자로서 실제 금융 인프라를 운영해본 경험에 기반할 때, 가장 현실적인 구조는 다음과 같다.

- **레이어 선택**: 이더리움 기반 L2(Polygon, Arbitrum 등) 또는 금융 특화 퍼블릭 체인. 메인넷 직접 사용은 비용·속도 문제로 부적합.
- **지갑 구조**: ERC-4337 기반 스마트 지갑. 단, guardian 복구 권한은 금융기관의 KYC 검증 기반으로 설계. 사용자는 소셜 로그인으로 접근, 백엔드에서 키 관리 서비스(KMS) 활용.
- **승인 체계**: 컨트랙트 레벨에서 거래 금액별 승인 단계 구현 (소액: 단순 서명, 고액: 다중 서명 또는 시간 지연). 은행의 내부 통제 원칙을 코드로 구현.
- **Paymaster 운영**: 금융기관이 직접 Paymaster를 운영하여 고객 가스비 대납. 수수료는 은행 수익 구조에 통합. Paymaster 장애를 대비한 fallback 로직 필수.
- **감사·모니터링**: 온체인 트랜잭션 + 오프체인 인덱서(The Graph 등) 조합으로 실시간 감사 로그 구성. 현행 FEP 로그 체계와 연동 가능한 구조 필요.
- **규제 대응**: 오프체인 KYC/AML 시스템과 온체인 지갑을 연결하는 화이트리스트 컨트랙트 설계. 거래 추적 가능성 확보.

### 5-3. 현재 금융권이 취할 수 있는 현실적 단계

단기적으로는 다음과 같은 단계적 접근이 현실적이다.

| 단계 | 내용 | 비고 |
|---|---|---|
| **1단계** | 내부 업무 자동화에 스마트컨트랙트 도입 (정산, 계약) | 프라이빗 체인으로 시작 |
| **2단계** | ERC-4337 기반 파일럿 지갑 서비스 (소규모 고객 대상) | L2 테스트넷 활용 |
| **3단계** | Paymaster + 커스텀 승인 정책 적용한 실 서비스 | KYC 연동 필수 |
| **4단계** | 타 금융기관 컨소시엄 체인과 상호운용성 확보 | BOK-Wire 연계 검토 |

중요한 것은 블록체인을 기존 시스템의 '대체'가 아닌 '보완'으로 접근하는 시각이다. 대외계 시스템에서 30개 이상의 기관과 AnyLink로 연결하듯, 블록체인도 기존 인프라와 공존하는 하이브리드 구조가 당분간 현실적이다.

#### 1. 해외 선제 도입을 통한 단계적 확장 전략

국내 금융 규제 환경에서 블록체인 기반 지갑 서비스를 곧바로 도입하는 것은 현실적으로 쉽지 않다. 전자금융거래법, 금융실명제, 외국환거래법 등 기존 규제 체계가 탈중앙화 지갑 구조를 직접 수용하기에는 아직 간극이 있기 때문이다.

이런 상황에서 현실적인 접근법은 **행내 글로벌서비스개발부를 통해 상대적으로 규제 유연성이 높은 해외 시장에서 먼저 가볍게 서비스를 오픈해보는 것**이다. 싱가포르, UAE, 홍콩 등은 디지털 자산과 Web3 서비스에 대한 규제 샌드박스 제도가 잘 갖춰져 있다. 해외에서 ERC-4337 기반 스마트 지갑 서비스를 소규모로 운영하며 실제 운영 노하우와 장애 대응 경험을 쌓는 것이 먼저다.

국내 규제는 가상자산이용자보호법 시행(2024년), CBDC 파일럿 사업 등을 통해 점진적으로 변화하고 있다. 규제가 완화되는 시점이 오면 이미 해외에서 검증된 아키텍처와 운영 경험을 바탕으로 국내 시장에 빠르게 대응할 수 있다. '규제가 열리면 그때 준비하자'는 수동적 태도보다, 해외 파일럿을 통해 기술과 조직 역량을 미리 갖춰두는 선제적 접근이 중장기 경쟁력 확보에 훨씬 유리하다.

## 6. 결론

EOA는 블록체인의 자기주권성을 가장 순수하게 구현한 방식이지만, 키 분실의 치명성과 제한된 기능으로 인해 대중 서비스에 적용하기엔 한계가 있다. ERC-4337은 그 한계를 스마트컨트랙트로 극복하며 은행 수준의 계정 관리 기능을 탈중앙화된 방식으로 구현한다.

트릴레마는 블록체인이 금융 인프라로 자리잡기 위해 반드시 해결해야 할 과제이며, L2 기술과 계정 추상화의 결합이 현재 가장 유력한 해법이다. 완전한 해결은 아직 요원하지만, 기술의 방향성은 명확하다.

주목할 점은, 이 기술들이 '블록체인의 기술적 진보'를 넘어 '금융 서비스의 접근성과 통제 구조를 어떻게 재설계할 것인가'라는 본질적 질문에 답하려 한다는 것이다. 새로운 기술이 도입될수록, 그것을 현업 맥락에서 해석하고 안전하게 운영할 수 있는 인력의 중요성도 함께 높아질 것이다.
